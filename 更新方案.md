下面是一份 “面向个人自用场景”的配置重构方案。目标是：

让用户第一次填一次 Cookie 就能开始用
默认隐藏“服务端化”“多设备协作”“远程推送”那套复杂配置
保留长期饭票内核能力，但不强迫普通用户理解它
一、重构目标
把当前配置体验从：

“你要理解 relay / client_id / shared_secret / ttl / refresh”

改成：

“先填一次完整 Cookie，插件自己尽量长期维持，失效了再补一次”

也就是：

默认面向个人自用
高级能力保留但折叠
长期饭票作为内部机制，而不是主配置概念

三、核心设计思路
1. 默认模式改成“自动托管长期票据”
不再让普通用户选择：

relay_ticket
manual_cookie_pool
对用户来说，默认就是：

插件会把你第一次填入的完整 Cookie 当成初始票据，然后自动接管、自动续命、自动持久化。

内部仍然可以继续用你现在的 relay_ticket 架构，但不要再把这个术语暴露成普通配置项。

2. 新增 bootstrap_cookie
这是整个重构的核心。

用户看到的含义
首次使用时，请粘贴一段完整的 Gemini / Google 浏览器 Cookie。
插件会自动提取必要认证信息并接管后续续期。
正常情况下你不需要反复修改，只有票据彻底失效时才需要重新填写。

插件内部行为
当 bootstrap_cookie 有值时：

标准化 Cookie
提取完整 cookie 视图
写入 auth_repo.sqlite 作为 active ticket
清理或忽略该字段的后续影响
后续全部走现有的 refresh / fallback / runtime reload 流程
也就是说：

bootstrap_cookie 是“引导启动用”
auth_repo.sqlite 才是长期真源
不是每次都读配置里的 Cookie
3. accounts 改成“高级兜底池”
保留，但不作为普通用户第一视角配置。

新的语义：

普通用户：可以完全不管它
高级用户：可以填多个备用 Cookie，用于 active ticket 彻底失效时 fallback
文案建议改成：

高级兜底账号池。
当主票据失效且自动刷新失败时，插件可临时切换到这里的备用 Cookie。
普通用户可留空。

五、_conf_schema.json 建议改法
1. 删掉普通用户视角下的 auth_mode
不要让用户再选：

relay_ticket
manual_cookie_pool
内部仍可保留该字段，但：

要么隐藏
要么由插件内部自动设定
要么仅高级模式可见
默认内部直接走“托管长期票据模式”。

2. 新增 bootstrap_cookie
建议类型：

text
建议文案：

标题

【3/4 首次认证】首次导入 Cookie
hint

首次使用时，请粘贴一段完整的 Gemini / Google 浏览器 Cookie。
插件会自动提取并接管后续认证。
正常情况下无需反复修改。
若后续票据彻底失效，再重新填写一次即可。
3. accounts 改为高级项
建议文案改成：

标题

【4/4 高级兜底】备用 Cookie 池
hint

可选。仅在主票据失效且自动续期失败时使用。
普通用户可留空。
每项填写一段完整 Cookie。
4. 高级 relay 项重新命名得更像“内部控制参数”
比如：

relay_shared_secret
改展示文案为：高级：外部票据推送密钥
relay_primary_client_id
改展示文案为：高级：允许推送主票据的设备标识
这样用户一看就知道：

这不是常规必填项
这是高级远程接入能力
六、运行时逻辑怎么调整
1. 插件启动时
如果检测到 bootstrap_cookie 有值：

自动导入 active ticket
写入 auth_repo.sqlite
生成完整 cookie_data
触发 runtime reload
然后后续就不再把它当主认证源。

2. 后续认证链路
后续继续沿用你现有设计：

active ticket
refresh
auth status
fallback
session 恢复
也就是说，这次重构主要是用户配置体验重构，不是底层架构推翻。

3. bootstrap_cookie 是否要自动清空
我建议：

不要自动清空用户配置项
但在逻辑上把它视为“初始化来源”，而不是“长期真源”
原因：

用户后续还可以看见自己曾经填过什么
出问题时也容易重新导入
清空反而可能让用户困惑
如果考虑安全，也可以后续再引入：

bootstrap_cookie_imported_at
或在导入后提示用户手动删除
但第一阶段不建议搞复杂。

这样更符合“个人 bot 插件”的真实认知。

八、推荐迁移策略
第一阶段：只改配置体验，不改底层能力
做这些就够了：

新增 bootstrap_cookie
隐藏或弱化 auth_mode
accounts 改成高级兜底池
README 改写成个人用户视角
保留原 relay 能力不删
第二阶段：增强自动导入体验
可以再补：

启动时检测 bootstrap_cookie 是否变化
若变化则重新导入 active ticket
/gemini_reverse doctor 显示“当前认证来源：bootstrap / relay / fallback”
九、最终建议的用户体验文案
建议对外口径统一成这句话：

首次使用时只需要填写一次完整 Cookie，插件会自动接管并尽量长期维持登录状态；只有当谷歌彻底判定票据失效时，才需要重新补一次。

这句话最符合：

个人自用
插件项目
不想理解太多内部机制
但又不虚假承诺“永不过期”
十、最终结论
这套重构的本质不是删掉长期饭票，而是：

把长期饭票从“用户必须理解的配置模型”改成“插件内部默认启用的认证机制”。

对你这个个人 bot 场景，我认为这是更合适的方向。
